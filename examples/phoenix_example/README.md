# Phoenix + Ltix Example

Minimal Phoenix app showing how to wire Ltix into an LTI 1.3 tool.

## Setup

```bash
cd examples/phoenix_example
mix setup
mix phx.server
```

The server starts at https://localhost:4000 (self-signed cert, generated by `mix phx.gen.cert`).

## LTI Routes

| Method | Path             | Description                              |
|--------|------------------|------------------------------------------|
| POST   | `/lti/login`     | OIDC login initiation (platform → tool)  |
| POST   | `/lti/launch`    | Authentication response (platform → tool)|
| POST   | `/lti/echo`      | Debug endpoint — dumps raw POST params   |

View all routes with `mix phx.routes`.

## How It Works

1. The platform POSTs to `/lti/login` with `iss`, `login_hint`, and `target_link_uri`
2. The tool calls `Ltix.handle_login/3`, stores the OIDC state in the session, and redirects to the platform's auth endpoint
3. The platform authenticates the user and POSTs back to `/lti/launch` with `id_token` and `state`
4. The tool calls `Ltix.handle_callback/3` to validate the JWT and extract launch claims
5. The launch template displays the parsed launch data (user, roles, context)

## Demo Registration

The app ships with a hardcoded registration in `lib/phoenix_example/lti_storage.ex`:

- **Issuer:** `https://platform.example.com`
- **Client ID:** `ltix-demo-client`
- **Deployment ID:** `deploy-001`

Update these values to match your LTI platform, or replace the in-memory storage with a database-backed implementation.

## Testing with curl

The echo endpoint is useful for inspecting what a platform sends:

```bash
curl -k -X POST https://localhost:4000/lti/echo \
  -d "iss=https://platform.example.com" \
  -d "login_hint=user-1" \
  -d "target_link_uri=https://localhost:4000/launch"
```
